## Q1：`_build_results_v001.json` 为何无法被 `moon clean` 清理

### 根因

`moon clean` 只删除 `_build/` 目录（moon 构建系统的编译产物目录）。  
`_build_results_v001.json` 是由如下命令产生的普通 JSON 文件：

```bash
bash run_local.sh _build_results_v001.json   # 脚本首参数 = 输出文件名
```

该文件不属于任何构建目标，moon 构建系统不知道它的存在，因此 `moon clean` 不会触碰它。

### 解决方案

```bash
rm _build_results_v001.json          # 手动删除
```

`.gitignore` 同步更新，覆盖此类文件：

```gitignore
results*.json
_build_results*.json
```

### 经验总结

`moon clean` 仅保证清理构建缓存，**不会**清理脚本产生的任意输出文件。  
建议将所有脚本输出文件加入 `.gitignore`。

## Q2：`moon run cmd/main` → `results.json` 工作流问题排查

### 问题描述

用户反映 v0.0.2 可以通过 `moon clean && moon build && moon run cmd/main` 生成 `results.json`，  
现在同样的命令无法生成。

### 根因分析

1. **`moon run cmd/main` 本身不生成文件**——输出到 stdout，需要通过管道提取 JSON 块。  
2. 正确工作流从 v0.0.2 起就是 `bash run_local.sh`，而非裸跑 `moon run`。  
3. 旧版 `main` 只运行 100 步（`nt_test = 100`），现版改为完整 `nt = 500` 步。

### 当前正确工作流

```bash
# 方式 A：推荐，一键生成（包含 Chorin + SIMPLE 双求解器）
bash run_local.sh                  # → results.json
bash run_local.sh my_output.json   # → 自定义文件名

# 方式 B：手工管道
moon run cmd/main --target wasm 2>&1 \
  | sed -n '/===JSON_DATA_START===/,/===JSON_DATA_END===/p' \
  | grep -v '===JSON_DATA' \
  > results.json
```

然后用浏览器打开 `cmd/main/local_viewer.html`，将 `results.json` 拖入指定区域即可可视化。

### JSON 提取机制

`main()` 在 stdout 输出 JSON 时使用标记分隔：

```
===JSON_DATA_START===
{ ... }
===JSON_DATA_END===
```

`run_local.sh` 用 `sed` 提取这段内容。浏览器查看器 `local_viewer.html` 也能识别这两个标记，可直接粘贴完整输出。

## Q3：SIMPLE 与 Chorin 求解器的灵活切换

### 求解器名称确认

非 SIMPLE 的求解器正式名称是 **Chorin 投影法**（Chorin Projection Method）：
- 算法：不可压缩 Navier-Stokes 显式时间推进 + 压力 Poisson 修正
- 提出者：Alexandre Chorin（1968）
- 特点：瞬态求解，每步显式更新，适合追踪流场演化过程

SIMPLE 为 Semi-Implicit Method for Pressure-Linked Equations（1972，Patankar & Spalding）：
- 特点：稳态迭代，欠松弛，适合求最终稳态解

### 三层切换机制

#### 1. 代码层——`main.mbt` 顶部常量（最灵活）

```moonbit
// ── 修改这些常量控制本地运行行为 ──────────────────────────

let run_chorin     : Bool = true   // false = 跳过 Chorin
let local_nt       : Int  = nt     // Chorin 步数（nt = 500）

let run_simple     : Bool = true   // false = 跳过 SIMPLE
let local_simple_n : Int  = 100    // SIMPLE 迭代次数

let timing_enabled     : Bool = true  // 计时总开关
let time_chorin_phase  : Bool = true  // Chorin 分项计时
let time_simple_phase  : Bool = true  // SIMPLE 分项计时
```

修改后重新运行 `bash run_local.sh` 即可。

#### 2. JSON 层——双求解器统一输出

`output_json()` 在同一 JSON 文件中输出两个求解器的结果：

```json
{
  "config": { "solver": "both", "nt": 500, "simple_iters": 100 },
  "timing": { "chorin_run_ms": 3882, "simple_run_ms": 789, "total_ms": 4672 },
  "statistics":       { ... },   // Chorin 统计（向后兼容键名）
  "grid":             [ ... ],   // Chorin 网格（41×41 = 1681 点）
  "simple_statistics":{ ... },   // SIMPLE 统计
  "simple_grid":      [ ... ]    // SIMPLE 网格（1681 点）
}
```

#### 3. 查看器层——浏览器 Tab 切换

`cmd/main/local_viewer.html` 在检测到 `simple_grid` 字段时，自动显示求解器选择按钮：

- **Chorin (projection, transient)** — 显示投影法结果
- **SIMPLE (pressure-correction, steady)** — 显示 SIMPLE 结果

切换时可视化（速度场、压力场）和统计数据面板同步刷新。

## Q4：计时系统清理与重写

### 旧代码问题

| 文件 | 问题 |
|---|---|
| `cmd/main/now_clock.c` | 基于 C DLL / Windows QueryPerformanceCounter，无法与 wasm-gc 配合 |
| `build_now_clock.ps1` | 构建该 DLL 的 PowerShell 脚本，依赖 MSYS2 gcc，环境复杂 |
| `fn now_seconds() -> Double { 0.0 }` | 存根永远返回 0.0，计时逻辑永不执行 |

**已删除**：`now_clock.c`、`build_now_clock.ps1`、`now_seconds()` 函数及全部调用。

### 新计时系统

使用 `moonbitlang/core/bench` 包提供的单调时钟：

```moonbit
// 底层 API（wasm 目标使用 Wasmtime 宿主导入）
@bench.monotonic_clock_start() -> Timestamp   // 记录起始时间戳
@bench.monotonic_clock_end(ts) -> Double      // 返回已过去的微秒数（µs）
```

封装为可配置的 `timed()` 帮助函数：

```moonbit
/// 执行 f()，测量并返回耗时（毫秒）。
/// enabled=false 时 f() 仍执行，但不打印、返回 0.0。
fn timed(enabled : Bool, label : String, f : () -> Unit) -> Double {
  if not(enabled) {
    f()
    0.0
  } else {
    let ts = @bench.monotonic_clock_start()
    f()
    let ms = @bench.monotonic_clock_end(ts) / 1000.0
    println("[Timing] \{label}: \{ms} ms")
    ms
  }
}
```

### 实测计时结果（41×41 网格，Wasmtime）

```
[Timing] Chorin init: 0.30 ms
[Timing] Chorin 500 steps: 3882 ms
[Timing] SIMPLE init: 0.08 ms
[Timing] SIMPLE 100 iters: 789 ms
=== Timing Summary ===
  Chorin init : 0.30 ms
  Chorin run  : 3882 ms
  SIMPLE init : 0.08 ms
  SIMPLE run  : 789 ms
  Total       : 4672 ms
```

计时结果同时嵌入 JSON 的 `timing` 字段，可在浏览器查看器的 **Timing** 面板中查看。

### 兼容性

| 运行目标 | 计时是否有效 |
|---|---|
| `moon run cmd/main --target wasm` | ✅ Wasmtime 提供宿主时钟 |
| `moon run cmd/main`（native） | ✅ 原生系统时钟 |
| `moon test --target wasm` | ✅（bench 包本身也使用同一接口） |
